E-Commerce Product Catalog Hierarchy

Java Full Stack Capstone - Module C (PS-29)

1. Project Overview

This project implements a polymorphic E-Commerce Product Catalog. It solves the problem of extensive instanceof type checks in procedural code by introducing an abstract Product class and specific subtypes (Physical, Digital, Subscription).

Recent updates have introduced Console Authentication, Role-Based Access Control, and CSV Data Persistence to make it a fully functional end-to-end application.

2. Directory & File Structure

Below is a clear breakdown of every folder and what the files inside them do:

src/com/ecommerce/model/ (Domain Entities)

  • Product.java: Abstract base class defining shared properties and calculation hooks (shipping, tax).
  • PhysicalProduct.java: Implements logic specifically for physical items (calculates shipping by weight).
  • DigitalProduct.java: Implements logic for digital items (instant download, digital tax).
  • SubscriptionProduct.java: Implements logic for subscriptions (recurring billing periods).
  • Order.java: Model representing a customer's purchase, storing their tracking ID, product ID, and cost.
  • User.java: Model representing a system user with a username, password, and role.

src/com/ecommerce/model/enums/

  • Region.java: Defines global regions (DOMESTIC, INTERNATIONAL, etc.) and shipping multipliers.
  • Role.java: Defines the security roles for users (ADMIN, CUSTOMER).

src/com/ecommerce/repository/ (Data Persistence)

  • ProductRepository.java: Handles saving and reading products to/from data/products.csv. Smartly reconstructs child classes based on the "Type" column.
  • OrderRepository.java: Handles saving and reading customer orders to/from data/orders.csv.
  • UserRepository.java: Handles saving and reading registered user accounts to/from data/users.csv.

src/com/ecommerce/service/ (Business Logic)

  • AuthService.java: Manages login, registration, and verifies user credentials against the UserRepository.
  • CatalogService.java: Manages adding products to the catalog and iterating through products polymorphically.
  • OrderService.java: Manages creating new orders and restricting order tracking to the specific user who purchased it.

src/com/ecommerce/util/

  • IdGenerator.java: Parses existing CSV files to automatically generate the next sequential IDs (e.g. P-001, TRK-001).

src/com/ecommerce/app/ (Entry Point)

  • EcommerceApp.java: The main entry point. Contains the interactive console menus, login prompts, and routes users to either the Admin Panel or Customer Portal.

data/

  • products.csv, orders.csv, users.csv: Flat text files serving as our database storage for permanent data persistence.

3. Security & Access Control

The application implements Role-Based Access Control (RBAC). The system generates a default Admin (admin / admin123). New users register as Customers.

  • Admin Portal: Can add new inventory, view full catalogs, and test locations.
  • Customer Portal: Can simulate purchases and track their own specific orders. Customers are blocked from viewing order histories belonging to other accounts.

4. Avoiding 'instanceof' with Polymorphism

By using the Template Method pattern in Product.calculateTotalCost(Region, String place), the catalog iterates purely on references of type Product. The JVM dynamically binds method calls to the specific subclass instance at runtime.

// Example of Polymorphism in OrderService
public Order createOrder(User user, Product product, Region region, String place) {
    // We don't care if it's physical or digital, Java figures it out!
    double totalCost = product.calculateTotalCost(region, place);
